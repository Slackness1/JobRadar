## 第 1 页｜项目概览

### 项目名称
**JobRadar**

### 我的角色
独立完成以下核心工作：
- **产品设计**：定义使用场景、筛选逻辑、公司追踪机制与信息展示结构
- **数据处理**：搭建多源抓取、字段清洗、去重、公司归一化与结构化入库流程
- **前后端开发**：完成数据库、后端 API、前端检索页、公司展开页与申请流程视图

### 项目目标
JobRadar 旨在解决求职中的三个高频问题：
1. **岗位信息分散**：官网、校招站、聚合站点分散，信息不完整；
2. **筛选成本高**：大量岗位噪声高，很难快速定位真正值得投递的机会；
3. **匹配标准不透明**：传统列表只能“看岗位”，无法解释“为什么这个岗位更值得投”。

### 当前项目亮点
- **多源岗位聚合**：整合历史数据与新增官网/校招站抓取结果；
- **公司定向重爬**：围绕目标公司池做持续追踪，而不是盲抓全网；
- **岗位评分机制**：基于赛道、关键词、技能等维度进行加权；
- **公司级视图**：同一家公司的岗位可按公司聚合查看，不再被分行/部门割裂；
- **外部情报整合（MVP）**：已打通岗位情报接口框架，可接入面经/待遇信息；
- **投递流程管理**：支持“已申请/已网测/一面/二面/三面”等状态管理。

## 第 2 页｜问题与产品思路

### 为什么原有求职方式效率低
- 聚合平台常常只给出公司名或简化岗位信息，**不一定对应真实可投入口**；
- 真正的申请入口往往在**公司官网或校招官网**，而且页面结构经常变化；
- 我的目标赛道并不是无限扩展，而是集中在少数核心方向和少数重点公司上；
- 因此，“全网盲抓”并不会提高决策质量，反而会增加筛选噪声。

### 产品思路的核心转变
JobRadar 的思路不是“尽可能多抓岗位”，而是：

> **围绕目标公司池做定向追踪，再用评分和情报把岗位变成可决策信息。**

具体来说：
1. 先建立目标公司池（例如银行、互联网大厂、重点城商行等）；
2. 优先打通官网/校招官网入口；
3. 对动态站点采用浏览器抓取，对稳定接口站点采用 API 抓取；
4. 用统一规则完成去重、归一化、评分和展示；
5. 在前端支持公司级追踪、状态管理和投递决策。

### 当前产品定位
从“岗位采集器”演进为：
- **目标公司追踪工具**
- **岗位匹配与筛选系统**
- **个人投递管理面板**

## 第 3 页｜系统功能

### 当前已实现的主要功能

#### 1. 多源岗位抓取
- 支持历史数据导入与新增官网/校招站抓取；
- 已打通多类站点：
  - 互联网公司校招站
  - 银行官网/校招站
  - 部分 API 型招聘站
  - 部分动态前端站点（Playwright 路径）

#### 2. 清洗与去重
- 统一字段：公司、岗位、地点、部门、岗位阶段、发布日期等；
- 去重逻辑基于 `job_id` 和详情页链接；
- 支持公司名归一化，例如把同一银行不同分行数据聚合到统一公司名下；
- 支持剔除社招，仅保留校招岗位。

#### 3. 数据库存储与更新
- 采用 SQLite 存储结构化岗位数据；
- 已支持 CSV/历史数据回补与增量同步；
- 已支持重新评分和公司级修复后的回写。

#### 4. 前端检索与筛选
- 岗位总览页支持：
  - 关键词搜索
  - 赛道筛选
  - 时间范围筛选
  - 岗位类型筛选（校招/实习/全部）
  - 最低分筛选
- 公司展开页支持：
  - 按公司查看全部岗位
  - 显示分行/部门信息
  - 继承主页面筛选或手动关闭继承
  - 当前筛选 / 全部数据 范围切换

#### 5. 投递与流程管理
- 支持岗位申请状态：
  - 待申请
  - 已申请
  - 已网测
  - 一面
  - 二面
  - 三面
- 已增加“**去掉已申请公司**”功能：
  - 若某公司已有岗位进入申请流程，可在总览中整体筛掉该公司。
- 已新增左侧独立栏目思路，用于集中查看“已申请及后续流程岗位”。

#### 6. 公司级重爬机制
- 支持对单个公司加入补爬队列；
- 适用于聚合站信息不全、官网更新延迟或目标公司优先维护场景。

#### 7. 岗位评分机制
- 按赛道、关键词、技能、地点、时效等维度加权；
- 支持前端展示总分与赛道分数；
- 支持结果解释与筛选联动。

#### 8. 外部情报整合（当前为 MVP）
- 已建立岗位情报模块与数据表；
- 已接入小红书情报线的最小可运行链路；
- 已支持登录态导入思路，但多平台深抓仍在推进中。

### 简化系统流程图（文本）
数据源 → 抓取器/API/浏览器 → 清洗归一化 → SQLite 入库 → 评分 → 前端检索/公司展开/申请流程 → 定向重爬与持续维护

## 第 4 页｜我的技术与业务贡献

### 技术层贡献

#### 数据抓取与结构化清洗
- 搭建多站点抓取框架，区分静态接口站与动态前端站；
- 对银行、互联网公司等不同类型站点编写定制化解析逻辑；
- 设计统一字段结构，完成去重、公司归一化、阶段过滤与历史数据回补。

#### 匹配规则与评分设计
- 设计岗位加权评分机制，把“是否值得投”变成结构化输出；
- 将赛道关键词、地点偏好、技能匹配、时效等因素组合到同一评分框架中；
- 保留推荐解释能力，避免纯黑盒排序。

#### 前后端开发与本地部署
- 实现岗位、评分、筛选、公司展开、重爬、情报等后端接口；
- 实现前端岗位总览页、公司展开页、状态管理与筛选逻辑；
- 完成本地部署与 SSH 隧道访问方案，支持在本机远程使用 VPS 上的系统。

#### 数据质量与流程控制
- 将社招与校招明确分离；
- 把分行/部门信息保留下来，但在公司层面聚合；
- 补充“去掉已申请公司”等面向真实投递流程的功能。

### 业务层贡献

#### 提高岗位筛选效率
- 从“多平台切换 + 手动比较”变为“单系统集中筛选”；
- 从“海量岗位浏览”变为“围绕目标公司池做重点追踪”。

#### 提升推荐透明度
- 让岗位不仅能被搜到，还能解释为什么值得关注；
- 把“岗位列表”升级为“带评分、带状态、带筛选继承”的决策界面。

#### 升级投递决策框架
- 把求职决策从“看单个岗位”升级为：
  **看公司 + 看匹配度 + 看流程状态 + 看外部情报**。

## 第 5 页｜结果与验证

### 当前阶段性结果
- GitHub 已有 **100+ Stars**；
- 系统已经形成完整的“**抓取 → 清洗 → 入库 → 评分 → 展示 → 申请管理**”闭环；
- 当前数据库中已累计到 **2 万级岗位数据**；
- 已实现银行与互联网两条核心线的实战推进；
- 已经可以直接用于个人求职场景中的：
  - 岗位检索
  - 重点公司追踪
  - 匹配排序
  - 投递状态管理

### 功能验证结果
- 已支持公司级展开查看全部岗位，并保留部门/分行信息；
- 已支持“当前筛选 / 全部数据”切换；
- 已支持申请流程状态更新与公司级排除；
- 已完成多批次实际岗位抓取与数据回补；
- 已能针对重点公司做“定向补抓”和修复。

### 业务价值验证
- 岗位信息不再停留在“采集”，而进入“可决策”的状态；
- 用户可以直接围绕核心公司池做高优先级投递；
- JobRadar 已经不是原型，而是一个能持续扩展的个人求职基础设施。

## 第 6 页｜反思与下一步

### 当前仍存在的限制
1. **部分站点反爬强**
   - 某些官网或校招站存在 TLS、反爬、动态渲染等问题；
2. **发布时间字段缺失**
   - 不少站点岗位日期字段不稳定，影响精细排序；
3. **情报模块仍偏 MVP**
   - 已有框架，但多平台真实抓取、评论提取和深度去重仍需加强；
4. **站点维护成本高**
   - 官网入口、接口参数、页面结构可能随时变化。

### 下一步优化方向

#### 1. 更强的官网入口自动发现
- 自动识别校招入口、岗位列表页和详情页；
- 降低人工维护站点配置的成本。

#### 2. 更稳定的目标公司追踪机制
- 完善第一档、第二档核心公司的持续更新；
- 对重点公司建立“优先修复/优先重爬”机制。

#### 3. 更细的评分与解释
- 引入更多结构化特征；
- 输出更明确的“建议投递 / 可观望 / 不建议投”的解释结果。

#### 4. 更完整的投递流程面板
- 把岗位筛选、公司排除、流程推进、日报输出进一步联动；
- 让系统更像一个求职运营面板，而不只是岗位列表。

#### 5. 外部情报能力升级
- 把小红书等平台从 MVP 推进到真实稳定抓取；
- 增强面经、待遇、工作节奏等外部信号对决策的辅助作用。

### 结论
JobRadar 当前已经完成了从“岗位聚合原型”到“个人求职决策系统雏形”的升级。它的核心价值不只是收集岗位，而是把目标公司、岗位匹配、申请流程与外部情报统一到一个可持续迭代的系统里。
